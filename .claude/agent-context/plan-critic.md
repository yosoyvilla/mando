### Risks Identified
- **apps/web is not a static SPA today** — it's a Nitro app (`vite.config.ts` has `nitro({ preset: "bun" })`; `package.json` has `"preview": "bun ./.output/server/index.mjs"` and existing server routes under `src/server/{claude,codex,opencode,system}`). Task 2.9 assumes "serving apps/web build output as static files with SPA fallback," and the Dockerfile task (6.1) assumes a plain static build artifact. Neither task touches `vite.config.ts` to remove the Nitro plugin or reconfigure the build to emit a pure static bundle. As written, `bun run --filter @mando/web build` still emits a server bundle (`.output/server` + `.output/public`), not a directory of static files Hono can serve with `serveStatic`. This is the single largest unaddressed gap — it will surface as a confusing build/deploy failure in Phase 6, far from where the real fix (removing/reconfiguring the Nitro plugin) belongs, which is Phase 5.
- **Root script mismatch will fail as literally written.** Root `package.json` currently has `"check-types": "turbo run check-types"` — there is no `"typecheck"` script. Task 0.1 Step 6 instructs "Run: `bun run typecheck`" and expects it to pass, but no step in Task 0.1 adds a root `"typecheck"` (or `"test"`) script to root `package.json`; only `turbo.json` tasks and `apps/web/package.json`'s own script are touched. `bun run typecheck` will fail at the root with "Missing script" before Task 0.1 is even complete. Same gap for a root `"test"` script (spec/README later reference `bun run test` conventions in Phase 7/8 CI).
- **Daemon exit-fast requires `.unref()`, not just `detached: true`** (see Doc Check #5). As written this will make `mando connect` block the parent shell until the daemon itself terminates — the opposite of the described UX and of `/mando`'s "shell injection stays snappy" requirement.
- **Zod major-version split across the workspace**: new packages pin `zod: "^3.24.1"`; `apps/web` already depends on `zod: "^4.4.3"`. Bun workspaces tolerate this, but it's an unexplained inconsistency the plan should either justify (e.g., "hub/protocol/agent are new and independently versioned, no cross-import of schemas with web") or fix by pinning to v4 throughout. Not a functional blocker since nothing in the plan shows web importing `packages/protocol`'s Zod schemas directly.
- **CI Postgres service container + Playwright's own `docker compose up -d postgres` in globalSetup are two different ways of getting Postgres**, and Task 8.3's CI workflow runs "Postgres (service container)" for unit/integration, then separately "Playwright E2E" whose `globalSetup` does its own `docker compose up -d postgres` migration. If CI already has a Postgres service container bound to 5432, and globalSetup's docker compose also tries to bind 5432, that's a port collision in CI. Not caught by the plan; needs either one Postgres source shared by both stages or distinct ports/env per stage.

### Recommended Changes
- Add an explicit sub-step to Phase 5 (before or inside Task 5.2) to either (a) remove `nitro({ preset: "bun" })` from `apps/web/vite.config.ts` and confirm `vite build` alone emits a static `dist/` directory Hono can serve, or (b) keep Nitro and have the hub proxy to/embed the Nitro server process instead of doing raw `serveStatic`. Pick one now — it changes Task 2.9 and Task 6.1's Dockerfile stage.
- Fix Task 0.1 to add root `package.json` scripts `"test": "turbo run test"` and `"typecheck": "turbo run typecheck"` (or rename `check-types` consistently) so Step 6's `bun run typecheck` and later CI references to `bun run test` actually resolve.
- Add `.unref()` to the Task 3.4 daemon-spawn step alongside `detached: true`.
- Add a note (README/AGENTS.md, Phase 7) documenting that `OPENCODE_SERVER_PASSWORD` on the local opencode server is unsupported in v1 (explicit non-goal) rather than silently ignored, or add Basic Auth passthrough to `forward.ts` if it's in scope.
- Replace fixed-timing SSE chunk assertions in Task 2.8 and E2E `session-drive.spec.ts` with order/content assertions to avoid CI flakiness.
- Reconcile the CI Postgres service container (8.3) with Playwright `globalSetup`'s own `docker compose up -d postgres` (8.1) — pick one Postgres instance per CI run or use distinct ports.
- Consider a small explicit "test server helper" task/interface (e.g., `apps/hub/test/helpers/testServer.ts` returning `{ url, close() }` via `Bun.serve({ fetch: app.fetch, websocket, port: 0 })`) introduced in Task 2.7 and reused by 2.8, 3.4, and Playwright, rather than leaving it implicit.

### Better Alternatives Considered
- Using `Bun.sql` (native) instead of `postgres.js`: spec explicitly deferred this to plan time; plan chose `postgres.js`, which is the safer choice for `sql.unsafe()` migration-running maturity and wider ecosystem docs — reasonable, keep it.
- Kustomize instead of plain manifests for k8s: user's own `~/.claude/rules/kubernetes.md` prefers Helm > raw manifests > Kustomize; plan uses plain manifests for a single Deployment/Service/Ingress/Secret, which is proportionate for this scope — acceptable deviation, not worth a Helm chart for one service.

### Missing Prerequisites
- Decision on apps/web build output shape (static vs Nitro-server) before Task 2.9/6.1 can be written correctly — currently an open question in the spec (#3) that the plan silently resolved to "static" without a task to make it true.
- Root `package.json` `test`/`typecheck` scripts before Task 0.1 Step 6 can pass.
- A CI-vs-Playwright Postgres reconciliation decision before Task 8.3 is written.
